# Day 18 – React State (`useState`)

## React State

Learned why normal JavaScript variables cannot be used to create dynamic user interfaces in React. Since components are functions, local variables are recreated every time the component executes, causing their values to reset.

React solves this problem using `useState`, which stores data outside the component and preserves it between re-renders.

```tsx
const [count, setCount] = useState(0);
```

`useState(0)` sets the initial state value only during the first render. On subsequent renders, React ignores the initial value and returns the latest stored state.

---

## Understanding `useState`

`useState()` returns an array containing two values.

```tsx
const [count, setCount] = useState(0);
```

Conceptually, React returns:

```tsx
[0, setterFunction]
```

JavaScript array destructuring assigns:

- `count` → Current state value.
- `setCount` → Function used to update the state.

The names `count` and `setCount` are chosen by the developer. React only returns the value and the setter function.

---

## Why `const` is Used

State variables are declared using `const` because they should never be modified directly.

```tsx
const [count, setCount] = useState(0);
```

Instead of writing:

```tsx
count++;
```

the state is updated using:

```tsx
setCount(count + 1);
```

The current variable never changes during the current render. React stores the new value internally and provides it during the next render.

---

## How `setCount()` Works

Calling the setter function does not directly modify the local variable.

```tsx
setCount(count + 1);
```

Instead, React:

1. Stores the new value internally.
2. Executes the component again.
3. Calls `useState()`.
4. Returns the updated value.
5. Updates only the changed part of the UI.

This allows React to manage rendering efficiently.

---

## State vs Normal Variables

A normal variable:

```tsx
let count = 0;
```

is recreated every time the component executes.

A state variable:

```tsx
const [count, setCount] = useState(0);
```

is stored by React outside the component, allowing the value to persist across renders.

## Emoji Changer Project

Implemented a simple React project to understand state management.

Created an array containing Unicode emojis and ASCII emoticons.

```tsx
const emojis = [
  "🤐", "💯", "😯", "😡", "😎", "😂", "😅", "😑", "😞",
  ";)", ":)", "^_^", "U_U", ":-)", "O.O", "~_~", "O_O",
  "😵‍💫", "🫡", "🤣", "🤗", "🤔", "😥", "😮", "😶‍🌫️",
  "🥱", "😪", "🤯", "🥵", "🤪", "😵", "😷", "😡",
  "🥴", "🤠", "🥸", "🥳", "🤬", "🙂‍↔️", "🤓", "🫣",
  "🙂‍↕️", "🧐", "🦄", "🐕", "🐽", "🐍", "🦔"
];
```

Stored the current emoji index using state.

```tsx
const [index, setIndex] = useState(0);
```

Displayed the current emoji dynamically.

```tsx
<h1>{emojis[index]}</h1>
```

Implemented a button to move to the next emoji.

```tsx
<button onClick={() => setIndex((index + 1) % emojis.length)}>
    Next Emoji
</button>
```

The modulo operator (`%`) resets the index to `0` after reaching the last emoji, creating a continuous loop through the emoji collection.

---

## Event Handling

Learned that `onClick` expects a function reference.

Incorrect:

```tsx
<button onClick={setIndex(index + 1)}>
```

This immediately executes `setIndex()` while rendering.

Correct:

```tsx
<button onClick={() => setIndex(index + 1)}>
```

or

```tsx
<button onClick={nextEmoji}>
```

This passes a function to React, which is executed only when the button is clicked.

---

## Hooks Rule

`useState` must always be called at the top level of a React component.

```tsx
function ChangeEmoji() {
    const [index, setIndex] = useState(0);
}
```

It should not be called outside a component, inside conditions, loops, or nested functions because React associates state with components only while they are rendering.
