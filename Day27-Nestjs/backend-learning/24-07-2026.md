# 📚 Designing a REST API

A REST API revolves around **resources** rather than actions.

For today's resource:

```
students
```

The REST endpoints are:

| HTTP Method | Endpoint | Purpose |
|-------------|----------|---------|
| GET | /students | Get all students |
| GET | /students/:id | Get a single student |
| POST | /students | Create a new student |
| PATCH | /students/:id | Update a student |
| DELETE | /students/:id | Delete a student |

This design keeps APIs clean, consistent, and easy to understand.

# 🧠 Thinking Before Coding

Before implementing the POST API, I learned to think about how real applications work.

For example:

When a user creates a student,

```json
{
  "name": "Swetha",
  "department": "ECE",
  "age": 20
}
```

the backend receives this data and processes it.

One important realization was that the **client does not generate the student ID**. In real-world applications, the backend or database is responsible for generating unique IDs.

# 📂 Updating the Service

Modified the `create()` method inside `students.service.ts`.

### Before

```typescript
create(createStudentDto: CreateStudentDto) {
  return 'This action adds a new student';
}
```

### After

```typescript
create(createStudentDto: CreateStudentDto) {
  return {
    message: 'Student created successfully',
    data: createStudentDto,
  };
}
```

---

# 📖 Understanding createStudentDto

The parameter

```typescript
createStudentDto
```

contains the data sent by the client.

For example, if the frontend sends:

```json
{
  "name": "Swetha",
  "department": "ECE",
  "age": 20
}
```

the same object is received inside the service.

Returning

```typescript
data: createStudentDto
```

allows the backend to work with dynamic user input instead of hardcoded values.

---

# 🧠 Why Not Hardcode the Data?

Instead of writing:

```typescript
return {
  name: "Swetha",
  department: "ECE",
};
```

the service returns the incoming request data.

This ensures that every user receives a response based on the data they submitted.
