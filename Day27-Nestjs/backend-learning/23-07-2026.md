# 📚 Generating a Resource

NestJS provides a CLI command that automatically creates all the required files for a resource.

```bash
nest g resource students
```

This command generated the following structure:

```
students/
│── dto/
│── entities/
│── students.controller.ts
│── students.module.ts
│── students.service.ts
```

Each file has its own responsibility, making the project modular and easier to maintain.

---

# 📁 Understanding the Generated Files

## students.module.ts

The module groups all the student-related files together.

It tells NestJS that the **StudentsController** and **StudentsService** belong to the same feature.

---

## students.controller.ts

The controller is responsible for handling incoming HTTP requests.

```typescript
@Controller('students')
export class StudentsController {
  constructor(private readonly studentsService: StudentsService) {}
}
```

### Explanation

```typescript
@Controller('students')
```

Creates the base route:

```
/students
```

Every route inside this controller will begin with `/students`.

---

### Constructor Injection

```typescript
constructor(
  private readonly studentsService: StudentsService,
) {}
```

Although I will learn Dependency Injection in depth later, I understood that this constructor allows the controller to use the methods available inside `StudentsService`.

---

# Creating the First GET API

```typescript
@Get()
findAll() {
  return this.studentsService.findAll();
}
```

### Explanation

```typescript
@Get()
```

Maps an HTTP GET request.

When someone visits:

```
http://localhost:3000/students
```

NestJS automatically calls:

```typescript
findAll()
```

Instead of writing the data directly inside the controller, it delegates the work to the service.

```typescript
return this.studentsService.findAll();
```

This keeps the controller clean and separates request handling from business logic.

---

# students.service.ts

The service contains the business logic.

Initially, the data is hardcoded.

```typescript
findAll() {
  return [
    {
      id: 1,
      name: "Uno",
      department: "CSE",
    },
  ];
}
```

### Explanation

The `findAll()` method returns an array containing student objects.

Since JavaScript objects are automatically converted into JSON by NestJS, visiting the endpoint returns:

```json
[
  {
    "id": 1,
    "name": "Uno",
    "department": "CSE"
  }
]
```

Currently, the data is manually written.

In future lessons, this method will retrieve data from a database instead of returning hardcoded values.

# 🛠 Debugging

While testing the API, I initially encountered an **ERR_CONNECTION_REFUSED** error.

To debug the issue, I verified:

- Controller mapping
- Module registration
- Application startup logs
- Route mapping
- Server status

After restarting the NestJS server correctly, the API responded successfully.

This helped me understand the difference between:

- **404 Not Found** → Server is running, but the requested route doesn't exist.
- **ERR_CONNECTION_REFUSED** → No server is reachable.
